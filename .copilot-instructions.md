# Copilot Code Style Guidelines

## C++ Version & Philosophy

**Prefer "Orthodox C++" coding style.** This means:

- **Target:** C++11/14 features are acceptable and encouraged where they improve clarity
- **Avoid:** C++17/20 features unless they significantly improve maintainability or readability
- **Examples of acceptable C++11/14:**
  - Smart pointers (UniquePtr, SharedPtr)
  - Range-based for loops
  - Lambda functions (especially for factory patterns)
  - EASTL containers
  - Strongly-typed enums (enum class)
  - Deleted/defaulted functions

- **Examples to avoid (unless justified):**
  - `if constexpr` (C++17) — **OK if it significantly reduces code duplication** (e.g., compile-time conditional branches in templates). Always ask before using.
  - `std::is_same_v<>` (C++17) — Generally use `std::is_same<>::value` (C++11), but `_v` variants are acceptable if already using C++17 features
  - Structured bindings (C++17) — use manual unpacking
  - Fold expressions (C++17)
  - Concepts (C++20)
  - Ranges library (C++20)

## Code Style Rules

1. **Naming Conventions:**
   - Class members: `myVariableName` (Hungarian-style prefix)
   - Constants: `UPPERCASE`
   - Functions: `CamelCase`
   - Types/Structs: `CamelCase`
   - Enums: `CamelCase`

2. **Formatting:**
   - Braces: Opening on same line, closing on new line (Allman-style)
   - Indentation: 2 spaces
   - Template syntax: spaces around `< >` (e.g., `ResourceHandle< T >`)
   - Smart pointers: `UniquePtr< T >`, `SharedPtr< T >`

3. **Comments:**
   - Comment why, not what
   - Use `//` for single lines, `/* */` for blocks (rarely needed)
   - Doxygen-style comments for public APIs

4. **Headers:**
   - Use `#pragma once` for include guards
   - Group includes: standard library, local utility, then project-specific
   - Forward-declare types to minimize includes

5. **Memory Management:**
   - Use `new`/`delete` for polymorphic types (as before)
   - Use smart pointers (UniquePtr, SharedPtr) where appropriate
   - Document ownership clearly (handles transfer vs. references)

6. **Template Usage:**
   - Use templates for generic containers/pools
   - Prefer specialization over `if constexpr`
   - Keep template code in headers
   - Document template parameters clearly

## Example: Conditional Template Logic

**DON'T (C++17):**
```cpp
template<typename T>
void Foo(T x) {
  if constexpr (std::is_same_v<T, int>) {
    // int case
  } else {
    // other case
  }
}
```

**DO (C++11 via specialization):**
```cpp
template<typename T>
struct FooImpl {
  static void Execute(T x) {
    // generic case
  }
};

template<>
struct FooImpl<int> {
  static void Execute(int x) {
    // int case
  }
};

template<typename T>
void Foo(T x) {
  FooImpl<T>::Execute(x);
}
```

## Why Orthodox C++?

1. **Readability:** Familiar patterns that most C++ developers understand
2. **Compatibility:** Works across older compilers and C++ standards
3. **Stability:** Fewer language features = fewer surprises
4. **Maintenance:** Smaller surface area for bugs and misunderstandings
5. **Consistency:** Matches the existing codebase style

## When to Break These Rules

- If a modern C++ feature **significantly** improves code clarity or safety
- If the feature is already heavily used in the codebase
- If you have explicit approval from the code owner
- Document the exception in a comment explaining why

## Future Revisions

As the codebase evolves, this document may be updated to reflect new decisions. Always ask the code owner before introducing new C++ standards or style patterns.
